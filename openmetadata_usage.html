How Synapse-Graph uses OpenMetadata

Mapping attention-based interpretability artifacts into a metadata control plane.
Tech stack

Topology mapping

graph LR M["Model (service)"] --> DB["Database (model)"] DB --> SC["Schema (transformer)"] SC --> L1["Layer_1 (table)"] SC --> L2["Layer_2 (table)"] L1 --> C11["Head_1 (column)"] L1 --> C12["Head_2 (column)"] L2 --> C21["Head_1 (column)"] C11 -. Tags .-> TAG["DEFECTIVE / QUARANTINED"] TAG --> API["Backend reads tags -> set_masked_heads()"]

Each transformer layer is represented as a table and each head as a column. Lineage edges connect prompt → layer.table → head.column → generated token.

Governance flow

  1. Topology bootstrap: backend creates or discovers synthetic OpenMetadata entities for model, layers, and heads.
  2. Trace ingestion: `AttentionTrace` objects are ingested as lineage edges linking prompts to table/column entities.
  3. Operator action: user tags a head column with `DEFECTIVE` or `QUARANTINED`.
  4. Runtime enforcement: backend polls/receives webhook, then calls `set_masked_heads()` to update `HeadMaskStore`. Masks take effect for subsequent generations (snapshot semantics).

Implementation notes: the code prefers column-level tagging via the OpenMetadata REST API and falls back to table-level tags when column ops fail. Tagging includes retry/backoff for robustness.

Querying and observability

Because heads are modeled as columns, data teams can query and filter by tags, view lineage for specific generations, and attach audit and ownership metadata to interpretable components.

This allows straightforward integration with existing data governance workflows (alerts, dashboards, and access controls) driven by familiar metadata primitives.
